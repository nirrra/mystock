# A股技术面分析框架设计

## 1. 背景与目标

本项目的目标是构建一个面向 A 股主板的技术面分析框架，用于日常筛选候选股票并输出可解释的分析结果。系统不涉及量化交易、自动下单或实盘执行，只服务于研究和分析。

第一阶段以命令行工具为交付形式，采用日线作为主筛选周期，分钟线作为辅助确认周期。系统需要支持三类策略模板：

- 趋势突破
- 回调低吸
- 强势股跟踪

系统需要在免费数据源可用的前提下先跑通整体流程，同时通过清晰的数据接口抽象，为后续切换到更稳定或更完整的付费数据源保留扩展空间。

## 2. 目标边界

### 2.1 目标

- 支持 A 股主板股票池构建与维护
- 支持主板股票的日线数据抓取、缓存与更新
- 支持基础技术指标计算
- 支持三类技术面策略模板的参数化筛选
- 支持输出候选股票列表、命中模板、关键指标和入选原因
- 支持后续接入分钟线作为二次确认层

### 2.2 非目标

- 不做自动交易、下单、回测撮合
- 不做实时盯盘系统
- 不做图形化 Web 界面
- 不以分钟线作为 V1 主筛选数据源
- 不在 V1 中引入机器学习排序或复杂评分模型

## 3. 研究问题重构

在免费数据源优先、命令行交付、不涉及交易执行的前提下，构建一个面向 A 股主板的技术面筛选框架。该框架以日线为主、分钟线为辅，面向短线和波段分析，支持趋势突破、回调低吸、强势股跟踪三类可配置策略模板，并输出候选股及可解释原因。

## 4. 核心假设

- 使用者主要关注短线和波段交易机会，对筛选结果的可解释性要求高于执行自动化程度。
- V1 的主要风险不在策略表达能力，而在数据获取稳定性和股票池定义是否准确。
- 免费数据源可以支撑 V1 的日线流程，但不应被视为长期唯一依赖。
- 分钟线在 V1 中只对日线初筛后的候选股进行辅助确认，不承担全市场主筛选职责。

## 5. 范围定义

### 5.1 股票池

V1 股票池仅覆盖沪深主板股票，默认排除以下对象：

- 创业板
- 科创板
- 北交所
- ST 或 *ST 股票
- 停牌股票
- 上市时间过短的新股
- 长期低流动性股票

新股和低流动性的具体阈值应配置化。默认建议：

- 新股定义：上市不足 60 个交易日
- 低流动性定义：近 20 日日均成交额低于预设阈值

### 5.2 数据周期

- 主周期：日线
- 辅助周期：5 分钟、15 分钟
- 1 分钟数据不纳入 V1 主流程

### 5.3 输出形式

命令行输出为主，同时可落地为 CSV 文件。单次运行的输出至少包含：

- 股票代码
- 股票名称
- 交易日期
- 命中策略模板
- 关键指标
- 入选原因

## 6. 架构设计

系统采用分层架构，避免将数据源、指标计算、筛选逻辑和结果输出耦合在同一个脚本中。

### 6.1 模块划分

- `src/data_sources/`
  - 负责对接数据源
  - 统一封装股票列表、日线、分钟线获取接口
- `src/universe/`
  - 负责股票池构建、清洗和过滤
- `src/indicators/`
  - 负责基础技术指标计算
- `src/screener/`
  - 负责将指标与模板规则组合成筛选逻辑
- `src/strategies/`
  - 负责定义趋势突破、回调低吸、强势股跟踪等参数化模板
- `src/reporting/`
  - 负责命令行展示和文件导出
- `config/`
  - 负责保存参数化规则、阈值和运行配置
- `data/`
  - 负责保存本地缓存数据

### 6.2 数据流

`数据源 -> 股票池清洗 -> 日线缓存 -> 指标计算 -> 策略模板筛选 -> 结果汇总 -> 命令行/CSV 输出`

分钟线只在日线初筛完成后介入：

`日线候选股 -> 分钟线拉取 -> 辅助确认 -> 最终输出`

## 7. 数据源设计

### 7.1 V1 数据源策略

V1 默认使用免费公开数据源实现整体流程，并在代码层引入统一的数据提供者接口。推荐以 AKShare 作为原型阶段主源，后续如对分钟线稳定性、覆盖范围或更新可靠性有更高要求，再切换或补充付费数据源。

### 7.2 数据源抽象接口

数据源层统一向上层暴露以下能力：

- `get_instruments()`
- `get_daily_bars(symbol, start, end)`
- `get_intraday_bars(symbol, freq, start, end)`

上层模块不得直接依赖第三方库的原始字段名和返回格式，避免替换数据源时大面积重写。

## 8. 数据存储设计

V1 采用本地文件缓存，不引入数据库。

### 8.1 存储格式

- 原始和清洗后的表格数据优先使用 Parquet
- 导出结果使用 CSV

### 8.2 逻辑表结构

#### instruments

- symbol
- name
- exchange
- board
- list_date
- is_st
- is_suspended

#### daily_bars

- symbol
- trade_date
- open
- high
- low
- close
- volume
- amount
- turnover

#### signals

- trade_date
- symbol
- strategy_name
- score 或 rank
- trigger_reason
- key_metrics

分钟线数据在 V1 中不做全市场长期归档，只对候选股进行近端临时拉取，降低存储和更新复杂度。

## 9. 指标体系

V1 基础指标限定为支持三类模板所需的最小集合：

- `MA5`
- `MA10`
- `MA20`
- `MA60`
- `20 日新高`
- `20 日均量`
- 单日涨跌幅
- 近 N 日回撤
- 波动率

如需要支持候选股排序，可在不改变框架的前提下增加相对强弱、量价扩张等派生指标，但不作为 V1 必选项。

## 10. 策略模板设计

策略不以硬编码脚本形式存在，而应以“模板 + 参数”的方式表达。模板定义结构应允许通过配置文件调整阈值，而无需修改 Python 逻辑。

### 10.1 趋势突破

典型条件示例：

- 收盘创 20 日新高
- 当日成交量大于 20 日均量的 1.8 倍
- 收盘站上 20 日和 60 日均线

### 10.2 回调低吸

典型条件示例：

- 60 日均线保持向上
- 最近 10 日最大回撤不超过配置阈值
- 回踩 20 日均线后重新放量

### 10.3 强势股跟踪

典型条件示例：

- 近 15 日涨幅位于全市场前部分位
- 近 3 日进入缩量整理
- 股价未跌破 10 日均线

## 11. 命令行设计

V1 命令行只保留最小必要入口：

- `update-universe`
- `update-daily`
- `screen --strategy breakout`
- `screen --all`
- `report --date YYYY-MM-DD`

命令职责应单一明确：

- `update-universe` 更新股票池
- `update-daily` 更新日线缓存
- `screen` 执行筛选
- `report` 输出指定日期结果

## 12. 错误处理

系统需要明确区分以下几类错误：

- 数据源连接或抓取失败
- 字段缺失或格式不一致
- 股票池构建失败
- 指标计算所需历史窗口不足
- 输出结果为空

处理原则如下：

- 数据源失败时记录日志并标记失败标的，不直接终止整批任务
- 历史窗口不足时跳过该股票的对应模板判断
- 无候选结果时输出空结果说明，而不是报错退出
- 所有关键阶段写入日志，便于后续排查

## 13. 测试策略

V1 测试聚焦于正确性而非性能极限。

### 13.1 单元测试

- 股票池过滤规则是否正确
- 各技术指标计算是否符合预期
- 模板条件组合是否按配置生效

### 13.2 集成测试

- 从数据抓取到结果输出的完整链路是否可跑通
- 当部分股票数据缺失时，流程是否可继续

### 13.3 人工验收

使用最近 3 到 6 个月的数据进行人工验收，重点检查：

- 候选股是否符合对“突破、回调、强势整理”的直觉判断
- 输出原因是否足够解释筛选结果
- 不同模板之间是否存在明显重复或逻辑冲突

## 14. V1 最小可交付范围

V1 完成标准如下：

- 能生成 A 股主板股票池
- 能更新日线行情并缓存到本地
- 能计算基础技术指标
- 能运行三类模板的第一版规则
- 能输出每日候选股列表及原因

未纳入 V1 的能力保持显式冻结，不在实现阶段临时扩张。

## 15. 实施顺序

推荐按以下顺序推进：

1. 固定股票池定义
2. 打通日线数据抓取和缓存
3. 实现基础指标
4. 实现三类模板的参数化规则
5. 打通命令行输出
6. 用近 3 到 6 个月数据做人审验收
7. 再补分钟线辅助确认

## 16. 技术选型

- Python 3.11+
- pandas
- Parquet
- YAML 配置
- logging
- Typer 或 argparse

V1 不要求引入数据库、消息队列、Web 服务或复杂指标库。若后续指标复杂度上升，再评估增量引入专用依赖。

## 17. 风险与缓解

### 17.1 免费数据源稳定性

风险：接口字段、限流或可用性变化会影响更新任务。

缓解：通过数据源抽象层隔离第三方依赖，保留后续替换空间。

### 17.2 股票池边界不清

风险：若主板过滤规则不严格，结果会混入非目标市场股票。

缓解：优先把股票池逻辑单独模块化并为其编写测试。

### 17.3 策略过早复杂化

风险：在 V1 中加入过多指标和评分逻辑，会掩盖基础数据和模板是否有效的问题。

缓解：严格限制基础指标和模板数量，先验证结果质量再扩展。

## 18. 结论

本项目的最佳起步方式是构建一个命令行驱动的 A 股主板技术面分析框架，以日线为核心数据层，三类参数化模板为核心策略层，并通过可解释输出支持日常选股分析。V1 先解决“能稳定筛出合理候选股”的问题，再逐步增强分钟线确认、数据源稳定性和结果展示能力。
