# Pattern 历史连续上涨门槛设计

## 1. 背景与目标

当前项目已经实现四类技术形态筛选，即 `pattern 1` 到 `pattern 4`。这些模式分别覆盖临近老高、平台突破、趋势回踩和二波启动，能够较好地描述“当前技术面看起来不错”的股票。

用户这次提出的约束不是再增加第五种模式，而是给现有四种模式统一加一层“历史弹性”门槛。目标很明确：排除那些当下技术图形好看，但过去较长一段时间内从未展示出短期连续上涨能力的股票。换句话说，入选股票不仅要有当前形态，还要在最近一段历史里证明自己曾经具备过短时间内快速走强的能力。

本次设计的目标是，在不改变四个 pattern 核心定义的前提下，为四个 pattern 增加一个统一、可配置、可测试的前置过滤条件。

## 2. 问题定义

当前四个 pattern 的判断逻辑主要基于均线结构、阶段涨幅、整理形态、放量突破和相对位置。它们能够回答“现在是否像一个可观察的技术形态”，但不能回答“这只票过去是否展示过足够的短期爆发性”。

这会带来一个实际问题：某些股票可能因为近期走势平稳、均线顺滑、位置合适而命中 pattern，但如果回看过去 200 个交易日，它从未在任意 5 个交易日内出现过超过 10% 的累计上涨，那么它的历史股性可能偏弱，后续即使形态成立，向上弹性也未必足够。

因此，系统需要一层统一过滤规则，用来表达下面这件事：

如果一只股票在最近 200 个交易日内，从未出现过任意一个 5 个交易日窗口累计上涨达到或超过 10%，则这只股票不应进入任何一个 pattern 的候选结果。

## 3. 目标边界

### 3.1 目标

- 为四个 pattern 增加统一的历史连续上涨门槛
- 默认口径为最近 `200` 个交易日、任意滚动 `5` 个交易日窗口、累计涨幅 `>= 10%`
- 规则通过配置文件管理，而不是硬编码在函数里
- 过滤逻辑在 pattern 公共入口生效，确保四个 pattern 一致
- 补足覆盖边界条件的自动化测试

### 3.2 非目标

- 不重写 `pattern 1` 到 `pattern 4` 的核心技术定义
- 不把这层历史门槛用于 TradingView 打分、MACD 背离判断或后续摘要文案
- 不引入更复杂的股性模型，例如连板史、涨停史、振幅分布或胜率统计
- 不在本次改动中新增 CLI 参数

## 4. 方案比较

### 4.1 在 `evaluate_strategies()` 前加统一过滤

做法是在公共策略入口先检查历史连续上涨门槛。只有通过该检查的股票，才继续进入 `type1` 到 `type4` 的原有判断。

优点：

- 四个 pattern 自动共享同一规则
- 没有重复代码
- 后续调整阈值时只改一处
- 行为最符合“共同准入门槛”的语义

缺点：

- 过滤逻辑和单个 pattern 实现分属两层，需要阅读入口才能看到全部条件

结论：

这是推荐方案。

### 4.2 在四个 `_apply_typeX()` 中分别加入条件

做法是在 `_apply_type1()` 到 `_apply_type4()` 内部各自复制一遍历史连续上涨判断。

优点：

- 单个函数自包含

缺点：

- 代码重复
- 后续调整阈值或口径时容易漏改
- 测试成本更高

结论：

不推荐。

### 4.3 在 `pattern` 结果导出前做后置过滤

做法是不改策略层，在 CLI 导出 CSV 前把不满足历史门槛的股票删掉。

优点：

- 看起来改动小

缺点：

- 规则只在某个输出链路生效
- 其他直接调用策略逻辑的路径会绕开过滤
- 语义上不属于“模式准入条件”

结论：

不采用。

## 5. 总体设计

本次采用“配置化公共前置过滤”的设计：

`history_df -> 历史连续上涨过滤 -> type1/type2/type3/type4 评估 -> 结果输出`

这意味着系统先问一个共通问题，再问四个 pattern 的各自问题。共通问题是：这只票在最近一段历史里，是否至少出现过一次足够强的 5 日连续上涨。只有答案为“是”，才允许进入后续的 pattern 判定。

## 6. 配置设计

需要为这层过滤新增独立配置结构，而不是塞进任意一个 pattern 配置中。原因是这不是 `type1` 的附属规则，也不是 `type2` 的特例，而是所有 pattern 共用的准入门槛。

建议新增配置块：

```yaml
history_momentum_filter:
  lookback_days: 200
  window_days: 5
  min_return: 0.10
```

对应的数据模型新增一个独立 dataclass，例如 `HistoryMomentumFilterConfig`，并挂载到应用级配置中。

这样做的好处是：

- 参数语义清晰
- 后续可以单独调整 `200`、`5`、`10%`
- 避免把公共规则错误归属到某一个 pattern 下

## 7. 策略入口设计

### 7.1 过滤位置

过滤逻辑应位于 `evaluate_strategies()` 入口，在任何 `_apply_typeX()` 被调用前执行。

预期行为是：

1. 先拿到单只股票完整历史数据 `history_df`
2. 执行历史连续上涨门槛检查
3. 如果不满足，直接返回空结果列表
4. 如果满足，再照常执行 `type1` 到 `type4`

这种放置位置最符合“统一准入门槛”的语义，也能保证未来无论启用哪些 pattern 组合，这层逻辑都不会被绕过。

### 7.2 判断口径

判断口径固定为以下规则：

- 只看最近 `lookback_days` 个交易日
- 在这段数据里按收盘价计算任意滚动 `window_days` 窗口累计涨幅
- 只要存在任意一个窗口的累计涨幅 `>= min_return`，就视为通过
- 如果一个满足条件的窗口都不存在，则视为不通过

这里的“5 天内上涨超过 10 个点”统一解释为“滚动 5 个交易日窗口收盘涨幅达到或超过 10%”。这个定义已经和用户确认，不再使用单日高点、盘中涨幅或前复权外的其他口径。

### 7.3 数据不足时的行为

如果可用历史数据不足以完成这项检查，系统应该采用保守处理方式，即不通过该门槛。

理由是：这条规则表达的是“历史上证明过自己”。当历史样本不足时，系统无法证明它满足条件，因此不应默认放行。

保守处理也与现有 pattern 体系一致，因为当前四个 pattern 本身已经对历史长度有最低要求。

## 8. 模块变更范围

### 8.1 `src/stocks_analyzer/models.py`

- 新增 `HistoryMomentumFilterConfig`
- 在 `AppConfig` 中挂载该配置结构

### 8.2 `config/default.yaml`

- 新增 `history_momentum_filter` 配置块
- 写入本次确认的默认值：`200`、`5`、`0.10`

### 8.3 `src/stocks_analyzer/strategies.py`

- 新增公共辅助函数，例如 `_has_recent_short_term_momentum_history()`
- 在 `evaluate_strategies()` 中先调用该函数
- 如果检查不通过，则直接返回空列表

这次改动不应修改 `_apply_type1()` 到 `_apply_type4()` 的模式定义本身。

## 9. 错误处理与可解释性

这次过滤是准入条件，不是异常场景，因此不需要额外报错。对于不满足条件的股票，系统只需要表现为“未命中任何 pattern”。

本次设计不强制要求把“不满足历史连续上涨门槛”的原因写入最终 CSV。原因是当前 `pattern` 输出只面向命中项，未命中项没有统一的失败原因导出结构。如果后续需要做全市场失败归因，再单独设计会更合理。

## 10. 测试设计

至少补充以下三类测试：

### 10.1 正常通过

构造一只股票，在最近 200 个交易日内至少存在一次滚动 5 日涨幅达到或超过 10%，同时其当前技术形态满足某个 pattern。预期结果是该股票不应被新的公共门槛挡掉。

### 10.2 最近 200 日内从未达标

构造一只股票，即使当前形态满足某个 pattern，但最近 200 个交易日内所有滚动 5 日窗口累计涨幅都低于 10%。预期结果是 `evaluate_strategies()` 直接返回空列表。

### 10.3 只有更早历史达标

构造一只股票，在 200 个交易日之外的更早历史里出现过 5 日涨超 10%，但最近 200 个交易日内没有。预期结果仍然是被排除，以验证时间窗口边界正确。

如果现有测试结构更适合直接验证辅助函数，也可以在函数级测试之外补一层集成测试，但至少要覆盖上述三种行为。

## 11. 风险与取舍

这条规则会让四个 pattern 的候选范围收缩，这是有意为之。它的目的就是排除“图形不错但历史弹性不足”的股票，因此候选数量减少不应被视为副作用，而应被视为规则生效的结果。

真正需要注意的风险有两个。

第一，某些长期稳健上行但单次 5 日涨幅不突出的股票，会因此被挡掉。这是这条规则与“历史弹性”目标之间的必然取舍。如果未来用户更偏好低波动慢牛票，可以通过配置下调阈值，而不是改变实现位置。

第二，若未来项目引入不同频率或不同复权口径，这条规则的定义需要保持统一，否则“5 日涨幅”会出现口径漂移。本次设计默认沿用当前 pattern 使用的同一份日线收盘价数据。

## 12. 验收标准

本次设计完成后的验收标准如下：

- 新增公共历史连续上涨配置，并成功从 `default.yaml` 载入
- 四个 pattern 共用同一层历史连续上涨门槛
- 最近 200 个交易日内不存在任意 5 日涨幅 `>= 10%` 的股票，会被统一排除
- 最近 200 个交易日内存在至少一次 5 日涨幅 `>= 10%` 的股票，不会因这层门槛被误杀
- 相关测试覆盖通过，并明确验证窗口边界
