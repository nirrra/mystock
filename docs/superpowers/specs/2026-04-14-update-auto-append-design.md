# Update 自动补全设计

## 1. 背景与目标

当前项目的 `update` 命令在日线缓存已存在时，已经具备“利用本地末尾日期做增量更新”的基础能力，但现有口径仍会把最后一个交易日重新拉一遍，再与本地缓存合并去重。这种做法适合把最后一天视为可能变化的数据，但不符合当前使用口径。

这次需求很明确：默认认为本地已缓存的数据是正确的，不再回刷最后一个交易日。`update` 在已有缓存时只负责向后补齐缺口，不负责修正末尾已有数据。

本次设计的目标是把 `update` 改成纯追加模式：

- 首次初始化时仍按 `--start-date -> --end-date` 全量拉取
- 本地已有缓存时，从“本地最后交易日的下一天”开始补全
- 如果本地最后交易日已经覆盖 `--end-date`，则直接跳过，不发请求

## 2. 问题定义

当前 `update` 的增量行为仍然包含“回刷最后一天”的语义：

1. 读取本地缓存
2. 找到本地最后一个 `trade_date`
3. 以最后日期当天作为远程请求起点
4. 合并新旧数据并按 `trade_date` 去重

这带来两个和当前需求冲突的点：

- 即使本地末尾已经正确，也会重复请求最后一天
- 即使本地已经覆盖 `--end-date`，仍会至少发起一次末尾刷新请求

用户这次明确要求把行为改成：

- 不新增新命令
- 继续使用现有 `update`
- 本地已有缓存时，只做向后补全
- 不允许用户通过更晚的 `--start-date` 人为制造日期缺口
- 不回刷最后一个交易日
- 本地已覆盖 `--end-date` 时直接跳过，不再发请求

## 3. 目标边界

### 3.1 目标

- 直接修改 `update` 的默认单股更新行为
- 本地不存在 parquet 时，仍按 `--start-date` 首次初始化
- 本地存在 parquet 时，自动从“本地最后日期的下一天”开始补全
- 合并新旧数据后按 `trade_date` 去重并升序写回
- 本地已覆盖到 `--end-date` 时直接跳过
- 批量更新时继续保持“单只失败不影响其他股票”

### 3.2 非目标

- 不新增 `append` 子命令
- 不改 provider 接口
- 不回刷最近 3 天
- 不回刷最后一个交易日
- 不改下游分析模块读取 parquet 的方式
- 不做分钟线缓存策略调整

## 4. 方案比较

### 4.1 直接修改 `update` 的单股更新逻辑

做法是在 `_run_update(...)` 的单股更新路径和批量循环路径中，统一改为“读取本地末尾日期 -> 以最后日期下一天为起点补全 -> 合并写回”；如果本地末尾已经覆盖 `--end-date`，则直接跳过。

优点：

- 改动集中
- 命令接口不变
- 最符合当前使用口径
- 对现有下游无侵入

缺点：

- `update` 的语义不再包含“末尾修正”

结论：

这是推荐方案，也是本次采用的方案。

### 4.2 保留末尾刷新，新增开关控制

做法是保留现有“回刷最后一天”的默认行为，再加一个开关决定是否纯追加。

优点：

- 兼容旧语义

缺点：

- 与当前需求不一致
- 增加使用成本
- 日常命令语义变复杂

结论：

不采用。

## 5. 总体设计

新的 `update` 行为分为两种情况。

### 5.1 首次初始化

当 `data/daily/<symbol>.parquet` 不存在时：

- 按用户传入的 `--start-date -> --end-date` 拉取完整区间
- 直接写入 parquet

此时 `--start-date` 完全生效。

### 5.2 已有本地缓存

当 parquet 已存在时：

1. 读取本地缓存
2. 取本地最后一个 `trade_date`
3. 计算自动起点 = `last_trade_date + 1 day`
4. 如果自动起点晚于 `--end-date`，直接跳过，不发请求
5. 从自动起点拉到 `--end-date`
6. 将远程增量数据与本地缓存合并
7. 按 `trade_date` 去重，保留新拉回的数据
8. 按 `trade_date` 升序写回 parquet

这里有两个关键约束：

- 一旦本地已存在缓存，`--start-date` 不再决定实际起点。即使用户传了更晚的日期，也不会跳过本地末尾之后的缺口。
- 自动起点固定为“最后日期下一天”，不是“最后日期当天”。`update` 被明确视为补缺口命令，而不是修正末尾已有数据的命令。

## 6. 参数语义调整

本次改动后，`update` 的参数语义需要明确解释为：

- `--start-date`
  - 仅在首次初始化时决定拉取起点
  - 本地已有缓存时不再生效
- `--end-date`
  - 继续表示本次更新的目标结束日期

这意味着用户对 `update` 的理解应变成：

- 第一次跑，用 `--start-date` 定义历史起点
- 以后再跑，系统会自动从最后一个交易日的下一天开始补齐后续缺口
- 如果本地已经覆盖到目标结束日期，本次更新会直接跳过

## 7. 合并与去重规则

为了保证 provider 偶尔返回重复行时缓存仍然稳定，合并规则固定为：

1. 拼接 `cached + fresh`
2. 统一将 `trade_date` 转为时间类型
3. 按 `trade_date` 去重，保留后出现的记录
4. 按 `trade_date` 升序排序
5. 重置索引后写回 parquet

默认假设单只股票的 `trade_date` 在日线数据里应唯一，因此按 `trade_date` 去重即可，不额外引入复合主键。

## 8. 模块变更范围

### 8.1 `src/stocks_analyzer/cli.py`

- 修改 `_run_update(...)`
- 为单只股票更新增加“自动起点计算 + 跳过判定 + 增量合并写回”逻辑
- 批量更新时复用同一套单股逻辑
- 在日志中明确输出：
  - 首次初始化
  - 自动补全起点
  - 已覆盖时跳过

### 8.2 `src/stocks_analyzer/storage.py`

本次不要求修改存储接口；合并与去重逻辑可以继续留在 `cli.py` 内部，只要实现集中且不在多个分支重复。

### 8.3 Provider 层

`baostock_provider.py` 和 `akshare_provider.py` 无需修改。

它们继续只负责：

- 接收起止日期
- 返回对应区间的数据

增量策略应留在上层，不应塞进 provider。

## 9. 错误处理与边界行为

需要固定以下行为：

- 本地 parquet 存在但为空：
  - 视为首次初始化
- 本地 parquet 存在但 `trade_date` 缺失或不可解析：
  - 记录 warning
  - 回退为按 `--start-date` 全量重拉该股票
- 本地最后交易日已经覆盖 `--end-date`：
  - 不发请求
  - 保留原文件
  - 记录“已覆盖，跳过”
- 增量请求返回空数据：
  - 不覆盖旧文件
  - 记录“无新增数据”
- 单只股票请求失败：
  - 批量模式下记录 warning 并继续其他股票

## 10. 测试设计

至少补充以下测试：

### 10.1 已有缓存时从最后日期下一天开始补全

构造本地 parquet 最后日期为某天，验证 provider 被调用时的 `start_date` 为“最后日期下一天”，而不是用户传入的更晚日期。

### 10.2 已覆盖结束日时直接跳过

构造本地最后日期已经覆盖 `--end-date`，验证不会发起任何请求。

### 10.3 首次初始化仍按 `--start-date`

本地无 parquet 时，验证 provider 仍按用户传入起点拉取。

### 10.4 合并后正确去重排序

构造本地数据和远程数据有重叠或重复日期，验证最终 parquet 只保留唯一日期且升序。

### 10.5 批量更新不因单只失败中断

延续当前批量容错测试思路，验证某只股票失败时其他股票仍继续更新。

## 11. 风险与取舍

第一，修改后 `update` 不再适合作为“末尾修正”命令。对当前日常使用来说这符合预期，但确实意味着它不能自动吸收供应商对最近一天的修正。

第二，本次设计不会回刷最后一个交易日，也不会回补更早几天。因此如果供应商历史修正发生在本地末尾或更早位置，缓存都不会自动修正。这是用户明确选择的行为结果，不是实现遗漏。

第三，`--start-date` 在已有缓存时失效，必须在 README 或帮助文案中讲清楚，否则用户会误以为它仍能控制增量起点。

## 12. 验收标准

本次设计完成后的验收标准如下：

- `update` 不新增子命令，直接沿用现有入口
- 本地已有缓存时，从“最后日期下一天”开始补全
- 不允许用户通过更晚的 `--start-date` 制造缺口
- 本地已覆盖 `--end-date` 时直接跳过
- 首次初始化仍按 `--start-date` 生效
- 合并后 parquet 去重、升序、可被现有分析链路继续读取
- 相关测试覆盖通过
