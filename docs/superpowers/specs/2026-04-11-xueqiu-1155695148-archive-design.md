# 雪球博主 1155695148 历史帖子 Markdown 归档设计

## 1. 背景与目标

当前项目主要围绕 A 股主板行情抓取、技术分析和报表输出，尚不具备社交内容采集能力。用户提出的新需求不是泛化的社区数据平台，而是一个明确、收敛的归档任务：

- 目标平台：雪球
- 目标账号：`1155695148`
- 目标内容：该账号的公开历史帖子
- 输出形式：单个 Markdown 归档文件
- 保留字段：发布时间、正文、原帖链接

该能力的用途是离线归档与后续人工阅读，不涉及情绪分析、结构化特征工程、自动摘要或数据库检索。

## 2. 问题定义

系统需要在没有雪球登录态、没有 `xq_a_token`、只依赖公开可访问页面的前提下，尽可能多地发现并抓取账号 `1155695148` 的历史帖子，将结果按时间倒序输出为一个 Markdown 文件。

这个问题的难点不在于 Markdown 渲染，而在于公开页抓取的不稳定性：

- 雪球存在反爬与访问限制
- 公开页面可能只暴露部分历史帖子
- 某些页面结构可能变化
- 搜索引擎收录不完整

因此该能力必须被定义为“最佳努力归档”，而不是“完整、稳定、可验证覆盖全部历史帖子”的数据源。

## 3. 目标边界

### 3.1 目标

- 仅支持固定账号 `1155695148`
- 尽可能发现该账号的公开帖子链接
- 提取每条帖子的发布时间、正文和原帖链接
- 按发布时间倒序生成单个 Markdown 文件
- 输出清晰的抓取统计与失败日志
- 为后续补充 Cookie 模式保留扩展空间

### 3.2 非目标

- 不做任意雪球用户的通用抓取器
- 不保证抓到账号全部历史帖子
- 不抓评论区、转发链、点赞用户或粉丝关系
- 不抓图片二进制文件，只保留正文文本
- 不做数据库入库和检索界面
- 不引入浏览器自动化作为 V1 默认方案

## 4. 方案比较

### 4.1 搜索引擎聚合

做法是通过搜索引擎检索 `site:xueqiu.com/1155695148/` 或类似模式，收集帖子链接后逐条解析。

优点：

- 实现成本最低
- 不依赖雪球内部接口

缺点：

- 搜索收录极不完整
- 结果受搜索排序和索引更新影响大
- 漏帖不可控

结论：

只能作为兜底入口，不能作为唯一发现机制。

### 4.2 公开页抓取器

做法是围绕该博主的公开页面与公开可访问帖子页，分成“链接发现”和“详情解析”两段执行。

优点：

- 架构最清晰
- 与当前项目的命令行工具形态兼容
- 后续可平滑补充 Cookie 模式

缺点：

- 无登录态时仍会遇到限流和缺页
- 页面结构变化会导致解析失效

结论：

这是 V1 推荐方案。

### 4.3 浏览器自动化

做法是使用 Playwright 或 Selenium 驱动真实浏览器滚动和提取内容。

优点：

- 对动态页面通常更稳

缺点：

- 依赖明显增重
- 维护成本高
- 对当前固定账号归档需求过度设计

结论：

V1 不采用，仅保留为后续补救手段。

## 5. 总体设计

V1 采用“公开页最佳努力抓取器”，流程拆成四步：

`帖子链接发现 -> 帖子详情抓取 -> 去重与排序 -> Markdown 渲染`

实现上新增一条独立命令，不与现有行情 `provider` 抽象耦合。原因是雪球帖子抓取和股票行情数据属于完全不同的数据域，强行复用 `DataProvider` 会造成接口污染。

推荐新增命令：

- `mystock xueqiu-archive`

该命令内部固定账号 `1155695148`，不向用户暴露通用 `user_id` 参数，避免误导为可泛化能力。

## 6. 模块设计

### 6.1 新增模块

- `src/stocks_analyzer/xueqiu_archive.py`
  - 负责公开帖子链接发现、详情抓取、去重和归档主流程
- `src/stocks_analyzer/xueqiu_rendering.py`
  - 负责 Markdown 文本渲染

如果实现过程中发现单文件职责过重，可以进一步拆成：

- `src/stocks_analyzer/xueqiu_client.py`
- `src/stocks_analyzer/xueqiu_parsing.py`

但 V1 不应为了抽象而过早拆分。

### 6.2 复用模块

- `src/stocks_analyzer/cli.py`
  - 新增 `xueqiu-archive` 子命令
- `src/stocks_analyzer/paths.py`
  - 新增雪球归档目录与缓存目录路径
- `src/stocks_analyzer/storage.py`
  - 如有必要，新增轻量文本或 JSON 缓存读写能力

## 7. 存储设计

雪球帖子不应与行情 parquet 混放。新增以下逻辑目录：

- `data/xueqiu/1155695148/discovered_urls.json`
- `data/xueqiu/1155695148/raw/`
- `reports/xueqiu/1155695148.md`

其中：

- `discovered_urls.json`
  - 保存已发现的原帖链接列表和去重后的链接键
- `raw/`
  - 可选地保存原始 HTML 片段或解析后的中间 JSON，便于调试
- `1155695148.md`
  - 最终归档结果

V1 可以不强制保留全部原始 HTML，但建议至少保留解析后的中间结构，方便后续页面结构变化时回归测试。

## 8. 数据模型设计

单条帖子在内存中的统一结构建议为：

- `post_id`
- `published_at`
- `url`
- `content_text`

字段要求如下：

- `post_id`
  - 从原帖链接中解析，作为去重主键
- `published_at`
  - 统一规范为 ISO 风格时间字符串，保留到分钟或秒
- `url`
  - 原帖完整链接
- `content_text`
  - 纯文本正文，去除主要 HTML 标签和多余空白

V1 不保留图片链接、点赞数、评论数、转发数等字段，因为用户已明确不需要。

## 9. 链接发现设计

### 9.1 发现策略

V1 采用多入口发现，按优先级执行：

1. 公开可访问的用户页面或帖子列表页
2. 公开帖子详情页中的关联链接
3. 搜索引擎结果作为兜底补充

发现器的职责只是尽量收集候选原帖链接，不负责正文解析。

### 9.2 链接过滤

仅保留满足以下条件的链接：

- 域名为 `xueqiu.com`
- 路径形如 `/{user_id}/{post_id}` 或其他明确指向单帖详情的格式
- `user_id` 必须为 `1155695148`

过滤阶段需要去掉：

- 用户主页
- 专栏聚合页
- 非目标账号链接
- 非单帖详情链接
- 重复链接

### 9.3 完整性声明

链接发现阶段必须在最终输出中保留声明：

- 当前归档来源于公开可访问页面
- 结果可能不覆盖全部历史帖子

## 10. 详情解析设计

对每个候选帖子链接执行详情抓取，并提取以下三项：

- 发布时间
- 正文
- 原帖链接

解析策略应尽量同时支持：

- 页面中的结构化元数据
- 正文区域 HTML
- 页面内可见发布时间文本

正文提取需要满足以下规则：

- 保留段落之间的自然换行
- 去除脚本、样式和明显导航文案
- 规整连续空白字符
- 避免把页面壳层说明混进正文

如果单条帖子缺少发布时间或正文，应记录警告并跳过，不将不完整数据写入最终归档。

## 11. Markdown 输出设计

输出文件固定为：

- `reports/xueqiu/1155695148.md`

文件结构建议如下：

1. 标题
2. 抓取说明
3. 统计摘要
4. 帖子正文列表

每条帖子按以下格式输出：

```markdown
## 2025-10-27 16:03

原帖链接：[https://xueqiu.com/1155695148/358401583](https://xueqiu.com/1155695148/358401583)

这里是清洗后的正文内容。
```

帖子之间保留稳定分隔，保证文件适合阅读和全文检索。

排序规则为发布时间倒序；如果发布时间相同，则按 `post_id` 倒序作为稳定次序。

## 12. CLI 设计

新增子命令：

- `mystock xueqiu-archive`

V1 建议保留少量实用参数：

- `--output`
  - 可选，自定义 Markdown 输出路径
- `--max-posts`
  - 可选，仅抓前 N 条，便于调试
- `--refresh`
  - 可选，忽略本地发现缓存并重新抓取

不增加 `--user-id` 参数，因为当前设计明确固定账号。

## 13. 错误处理设计

该功能要接受网络和页面不稳定现实，因此采用“单条失败不阻塞全量任务”的策略：

- 发现入口失败时，记录日志并尝试下一个入口
- 单条帖子解析失败时，记录失败链接并继续
- 最终只要成功解析到至少一条帖子，就生成归档文件
- 如果一条都未成功解析，则命令以失败状态退出

日志中至少输出：

- 发现到的候选链接数
- 去重后的链接数
- 成功解析的帖子数
- 失败链接数
- 最终输出文件路径

## 14. 测试设计

由于公开网页不稳定，测试不能强依赖实时网络。V1 测试以纯函数和样例输入为主：

- 链接过滤测试
- 帖子 ID 解析测试
- 发布时间解析测试
- 正文清洗测试
- Markdown 渲染测试

如果实现中引入中间 JSON 结构，还应补一个端到端小样例测试：

- 输入固定的帖子对象列表
- 输出稳定的 Markdown 文本

不建议在默认测试集中加入真实联网抓雪球的集成测试，以免测试结果依赖外部站点状态。

## 15. 风险与后续扩展

### 15.1 主要风险

- 公开页结构变化导致解析失效
- 无登录态导致能见范围有限
- 限流或反爬导致抓取不稳定

### 15.2 缓解手段

- 保持发现与解析分层，降低修复成本
- 在日志和归档头部明确“最佳努力，不保证完整”
- 为后续增加 Cookie 配置预留接口

### 15.3 后续可扩展方向

- 增加可选 Cookie 模式，提高覆盖率
- 从固定账号扩展到可配置账号列表
- 追加图片链接、互动数等元数据
- 将归档内容转换为结构化训练语料

## 16. 结论

V1 最合适的实现方式是在现有项目中新增一条独立的 `xueqiu-archive` 命令，固定抓取雪球账号 `1155695148` 的公开历史帖子，输出一个按时间倒序的 Markdown 归档文件。

该设计刻意收窄范围，不把它包装成通用社交数据平台，也不承诺完整覆盖。这样可以在当前无登录态的现实约束下，以较低复杂度交付一个可用的离线归档能力，并为后续更稳定的 Cookie 模式保留升级空间。
